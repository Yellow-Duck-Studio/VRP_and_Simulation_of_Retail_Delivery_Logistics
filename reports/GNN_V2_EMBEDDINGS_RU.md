# Изменение embeddings в существующей GNN

## 1. Что изменено

Изменён только способ, которым существующая GNN строит embedding заказа.

Мы не создаём новый пайплайн и не заменяем алгоритм команды. Новая логика находится
внутри существующего класса `ClusteringGNN` в файле `GNN/model.py`.

Обычный вызов модели остался прежним:

```python
logits = model(data)
```

Поэтому текущие обучение, predict и сборка кластеров продолжают вызывать модель так же,
как раньше.

## 2. Что не менялось

В этой задаче не изменялись:

- входные признаки и построение графа в `GNN/data.py`;
- формула и расчёт cost в `GNN/costs.py`;
- формат обучающих данных;
- функция потерь и цикл обучения в `GNN/train.py`;
- сборка кластеров и repair в `GNN/decode.py`;
- predict и infer;
- проверки итоговых кластеров.

Таким образом, новый encoder можно соединить с актуальными решениями команды без
переноса cost или кластеризации из отдельной версии.

## 3. Общая схема

```text
Текущий граф команды  ->  Новый encoder embeddings  ->  Текущий affinity head  ->  Текущий decode и сборка кластеров
```

Меняется только второй блок.

## 4. Как embedding строился раньше

Сначала признаки заказа переводились во внутренний вектор размерности 128. Затем он
проходил через четыре слоя `TransformerConv`.

Embedding был просто результатом последнего слоя. Из-за этого могли теряться исходные
локальные признаки заказа: последний слой в основном содержал уже смешанную информацию
обо всех соседях.

Также итоговый embedding не имел постоянной длины. Его числовой масштаб мог меняться
между задачами и эпохами обучения.

## 5. Как embedding строится теперь

### Нормализация входа

Перед первым слоем отдельно нормализуются:

- признаки заказов;
- признаки связей между заказами.

Для этого используется обучаемая `BatchNorm1d`. Она помогает модели работать с
признаками разного масштаба: километрами, минутами и массой.

Сами признаки и их смысл не изменяются.

### Более плавное преобразование признаков

После нормализации признаки проходят через небольшие блоки:

```text
Linear -> GELU -> LayerNorm
```

Это создаёт начальное представление заказа и ребра размерности 128.

### Использование всех GNN-слоёв

Раньше использовался только результат последнего слоя.

Теперь сохраняются:

- начальное представление заказа;
- результат первого GNN-слоя;
- результат второго слоя;
- результат третьего слоя;
- результат четвёртого слоя.

Модель сама обучает пять весов и решает, сколько информации брать с каждого уровня.

Это похоже на `Jumping Knowledge`: embedding одновременно хранит локальную информацию
о заказе и контекст от ближайших и более дальних связей графа.

### Финальная проекция

Смешанное представление проходит через отдельный projection head. После этого embedding
нормализуется по L2:

```text
длина embedding = 1
```

Размер embedding остаётся равным 128. Поэтому остальным частям системы не нужна новая
настройка размерности.

## 6. Почему это может работать лучше

Новый подход решает три проблемы старого encoder.

1. Модель меньше зависит от исходного масштаба признаков.
2. Последний GNN-слой больше не может полностью затереть исходную информацию о заказе.
3. Нормализованные embeddings проще сравнивать между собой.

При этом модель по-прежнему использует те же признаки, labels и эталонные кластеры,
которые подготовила команда.

## 7. Изменение affinity head

Affinity показывает, насколько два заказа подходят для одного кластера.

Раньше в head передавались `embedding_A`, `embedding_B` и их разница. Такой порядок не
был полностью симметричным: теоретически результат для `A, B` мог отличаться от
результата для `B, A`.

Теперь используются только симметричные комбинации:

```text
embedding_A + embedding_B
abs(embedding_A - embedding_B)
embedding_A * embedding_B
существующие признаки ребра
```

Перестановка заказов не меняет эти значения. Формат выходных logits при этом остался
прежним.

## 8. Как получить embeddings напрямую

Обычный pipeline продолжает использовать:

```python
logits = model(data)
```

Если алгоритму сборки нужны сами embeddings, их можно получить без запуска отдельной
модели:

```python
embeddings = model.encode(data)
```

Результат имеет форму:

```text
количество заказов x 128
```

Это позволяет Ване использовать embeddings в своём актуальном сборщике кластеров, не
перенося туда старую или экспериментальную реализацию decode.

## 9. Обратная совместимость

Совместимы:

- сигнатура `ClusteringGNN`;
- вызов `model(data)`;
- размер выходного массива logits;
- текущие `train.py`, `predict.py`, `infer.py` и `decode.py`;
- текущий формат графа.

Старые веса модели несовместимы с новым внутренним устройством encoder. Модель нужно
обучить заново. Это ожидаемо: изменились обучаемые слои, хотя внешний интерфейс остался
тем же.

## 10. Запуск

Используется существующая команда обучения. Нового `train_v2.py` нет.

```bash
python GNN/train.py \
  --warehouses data/large/warehouses.csv \
  --orders data/large/orders.csv \
  --transport data/transport_types.csv \
  --solutions data/large/ilp_master.json \
  --out GNN/model_embeddings.pt
```

Predict тоже запускается существующей командой:

```bash
python GNN/predict.py \
  --warehouses data/large/warehouses.csv \
  --orders data/large/orders.csv \
  --transport data/transport_types.csv \
  --model GNN/model_embeddings.pt \
  --out predictions.json
```

Все cost и итоговые кластеры в predict считаются текущей командной реализацией.

## 11. Что нужно сравнить в эксперименте

Старый и новый encoder нужно обучить на одинаковом train/validation split и сравнить:

- validation loss;
- edge precision, recall и F1;
- итоговый cost после актуального сборщика команды;
- долю валидных кластеров;
- нарушения SLA;
- время работы.

Главная проверка — улучшает ли новый embedding итоговые кластеры команды. Сам по себе
красивый embedding ещё не гарантирует меньший cost.

## 12. Какие файлы относятся к изменению

- `GNN/model.py` — новый способ построения embeddings.
- `GNN/test_embeddings.py` — проверка интерфейса, L2-нормы и симметрии.
- `reports/GNN_V2_EMBEDDINGS_RU.md` — этот отчёт.

Остальные части GNN не переписывались в рамках исправления.
