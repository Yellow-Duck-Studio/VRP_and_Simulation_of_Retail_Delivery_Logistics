# Точный ILP-решатель кластеризации заказов по курьерам

**Точный** (не эвристический) решатель, который разбивает заказы одного
склада на курьерские рейсы (кластеры), минимизируя суммарную стоимость
доставки с учётом временных окон, грузоподъёмности транспорта и
ограничения на размер парка курьеров.

Это масштабируемая замена решателю на битмасковом ДП: гарантия
оптимальности та же самая, но на практике решатель масштабируется
дальше. Если говорить точно о сложности:

- **Этап 1 (перебор кандидатов)** при фиксированном `max_k` честно
  полиномиален по `n`: число кандидатов равно
  `Σ_{k=1}^{max_k} C(n, k) · k!`, то есть `O(n^max_k)`, а не `O(2^n)`.
  Именно это убирает экспоненциальный взрыв, который есть у полного
  битмаскового ДП (оно хранит состояние для всех `2^n` подмножеств
  заказов склада, что нереально уже примерно при n ≈ 20).
- **Этап 2 (ILP)** — это set-partitioning, которая в общем случае
  **NP-трудна**: branch-and-cut в `scipy.optimize.milp` (HiGHS) не даёт
  полиномиальной гарантии в худшем случае. На практике он масштабируется
  дальше, чем брутфорс-ДП на битмасках, потому что число переменных — это
  число *кандидатов-кластеров* (а не `2^n` подмножеств заказов), а
  структура ограничений (exact cover + concurrency по интервалам) хорошо
  ложится на branch-and-cut — HiGHS обычно решает задачи с тысячами
  бинарных переменных за секунды. Но это эмпирическое/практическое
  преимущество, а не асимптотическое.

Честный итог: этап 1 меняет экспоненту на полином (ценой ограничения
размера кластера сверху параметром `max_k`), а этап 2 остаётся NP-трудным
теоретически, но достаточно быстрым на практике для реалистичного числа
заказов на склад (~30). Гарантии полиномиального времени в худшем случае
для всего пайплайна в целом нет.

## Как это работает

### 1. Перебор кандидатов-кластеров (векторизованный брутфорс)

Для каждого склада рассматриваются все подмножества заказов размером
`1..max_k` (по умолчанию `max_k=5`) как кандидаты-кластеры. Для каждого
подмножества перебираются все порядки объезда (перестановки) и все типы
транспорта, чтобы найти самый дешёвый допустимый маршрут — это тот же
самый брутфорс-расчёт стоимости, что и в обычном решателе на битмасках,
только посчитанный сразу для всех подмножеств/перестановок/типов
транспорта через NumPy вместо циклов на Python — именно это даёт
необходимую скорость на больших объёмах.

Для каждого кандидата-кластера решатель считает:

- **Расстояние/стоимость маршрута туда-обратно**, включая обратный путь
  от последней точки доставки до склада (курьеру физически нужно
  вернуться — раньше это считалось только «в одну сторону», что занижало
  стоимость кластеров).
- **`start_epoch` / `finish_epoch`** — момент, когда курьер выезжает со
  склада, и момент, когда он возвращается и освобождается для
  следующего рейса.
- **Допустимость** по временным окнам (готовность к отгрузке/дедлайн)
  каждого заказа и по грузоподъёмности выбранного транспорта.
- **Стоимость** по модели ниже.

Недопустимые кластеры отбрасываются, остаются только допустимые.

### 2. ILP на set-partitioning (`scipy.optimize.milp`, HiGHS)

Из всех кандидатов-кластеров целочисленная программа выбирает подмножество
так, чтобы:

- каждый заказ был покрыт **ровно один раз** (ограничение
  set-partitioning), и
- суммарная стоимость была минимальна.

Решается точно через `scipy.optimize.milp` (бэкенд HiGHS, идёт в
комплекте со SciPy — внешний решатель или сеть не нужны). Это заменяет
битмасковое ДП с `2^n` состояниями на ILP-формулировку той же самой
задачи, поэтому результат остаётся доказуемым оптимумом, а не
приближением.

### 3. Ограничение на размер парка курьеров

В реальности парк курьеров конечен, особенно в пиковую нагрузку. ILP
дополнительно требует, чтобы одновременно «в полёте» (`start_epoch <= t <
finish_epoch`) было не более `max_couriers` выбранных кластеров — это
реализовано одним линейным ограничением на каждую точку разрыва по
времени (sweep-line по уникальным моментам старта). Курьер, закончивший
рейс A до начала рейса B, свободен для рейса B — ограничение лимитирует
именно одновременную занятость, а не общее число рейсов.

Так «самое дешёвое разбиение при бесконечном числе курьеров» превращается
в «самое дешёвое разбиение, достижимое ограниченным парком курьеров».

Если при заданном `max_couriers` задача несовместна, решатель один раз
дополнительно решает её без ограничения на курьеров (только для
диагностики) и сообщает, сколько курьеров потребовалось бы для самого
дешёвого разбиения без ограничения.

`max_couriers` при генерации обучающих данных сэмплируется равномерно для
каждой пары (задача, склад) из диапазона `COURIER_M_RANGE` (по умолчанию
`(2, 15)`), чтобы модель ниже по пайплайну (например, GNN) видела примеры
с разными размерами парка и училась учитывать этот параметр, а не
воспринимала размер парка как зашитую константу одного датасета.

## Модель стоимости

Для кластера, обслуживаемого транспортом `t` с суммарной массой
`mass_combo`:

```
cost = fixed_fee
     + per_km_fee    * общее_расстояние_туда_обратно_км
     + per_order_fee * количество_заказов
     + per_kg_min_fee * сумма_по_перегонам(минуты_перегона * оставшаяся_масса_кг)
```

Компонент «кг·мин» взимается только за массу, которую курьер **всё ещё
везёт** на данном перегоне — оставшаяся масса уменьшается после каждой
доставки, а не остаётся равной полной массе кластера на всём маршруте.

## Формат входных данных

**`orders.csv`**

| колонка | тип | примечание |
|---|---|---|
| `task_id` | str | |
| `warehouse_id` | str | |
| `order_id` | str | |
| `order_lat`, `order_lon` | float | |
| `pickup_ready_at` | datetime | |
| `delivery_deadline_at` | datetime | |
| `total_mass_kg` | float | |

**`warehouses.csv`**: `task_id`, `warehouse_id`, `lat`, `lon`.

**`transport_types.csv`**: `code`, `fixed_fee`, `per_km_fee`,
`per_order_fee`, `per_kg_min_fee`, `max_payload_kg`, `approx_speed_kmh`.

ID-колонки (`task_id`, `warehouse_id`, `order_id`) принудительно
приводятся к `str` при загрузке — иначе pandas молча повышает их до
`float64`, если в колонке встречается `NaN` (`"1"` → `1.0`), что портит
строковые ключи, записываемые в архивы ниже по пайплайну.

Даты приводятся к эпохе в секундах устойчиво к тому, какое разрешение
datetime64 выберет pandas (`ns`/`us`/`ms`/`s` отличается в разных версиях
pandas и окружениях).

## Выходные данные

Вызов `main(...)` создаёт три файла (с префиксом `out_prefix`):

- **`{out_prefix}_clusterizations.json`** — полное решение по каждой паре
  задача/склад (выбранные кластеры, стоимость, транспорт, тайминг).
- **`{out_prefix}_summary.csv`** — по одной строке на пару (задача,
  склад): число заказов, `max_couriers`, число кластеров, суммарная
  стоимость, статус (`ok` / `infeasible`).
- **`{out_prefix}_master.json`** — архив в стиле
  `master_clusterizations.json`, сгруппированный по складам (ID заказов
  уникальны только внутри задачи, а не глобально), где каждый склад
  отображается на `{"clusters": [...], "max_couriers": M}`, чтобы пайплайн
  обучения мог восстановить, при каком ограничении на парк курьеров было
  построено разбиение.

## Использование

```python
from brute_force_ilp import main

results, summary_df = main(
    orders_path="data/orders.csv",
    warehouses_path="data/warehouses.csv",
    transport_path="data/transport_types.csv",
    out_prefix="ilp",
    max_k=5,                 # максимальный размер кластера
    courier_m_range=(2, 15), # None отключает ограничение на парк курьеров
    seed=42,
)
```

Либо из командной строки (отредактируйте пути в конце скрипта или
адаптируйте блок `if __name__ == "__main__":`):

```bash
python brute_force_ilp.py
```

## Замечания и ограничения

- Сложность первого этапа растёт комбинаторно с ростом `max_k` и числа
  заказов на склад; параметр `max_k` ограничивает размер кластера, чтобы
  перебор оставался практичным (по умолчанию `5`).
- Решение остаётся **точным оптимумом** для сформулированной ILP-задачи —
  это не эвристика, единственное ограничение — потолок `max_k` на размер
  кластера.
- Ограничение на одновременную занятость задаёт именно *ёмкость парка*, а
  не конкретное назначение «курьер → рейс» (что добавило бы ILP лишнюю
  симметрию без пользы); функцию `required_couriers()` можно использовать
  отдельно, чтобы посчитать минимальное число курьеров, необходимое для
  выполнения произвольного заданного списка рейсов.
